# Design Note (ADR) 작성 스킬

이 저장소의 평가 핵심은 **"코드의 설계 의도를 설명할 수 있는가"** 다. 이 스킬은 비자명한 결정을
일관된 ADR(Architecture Decision Record) 형식으로 `docs/design-notes/` 에 남긴다.

## 언제 쓰는가
- 라이브러리/패턴/구조 수준의 선택을 했을 때 (예: 라우팅 방식, 상태관리 경계, 빌드 구성)
- 트레이드오프가 있는 결정 (단순함 vs 유연함, 성능 vs 가독성 등)
- 스펙에 없어 "합리적 기본값"으로 정한 동작
- 인라인 주석으로 담기엔 큰 맥락이 필요한 경우

(작은 지역 결정은 ADR 대신 코드 인라인 "설계 의도" 주석으로 충분하다.)

## 절차
1. `docs/design-notes/` 에서 가장 큰 번호를 확인하고 다음 4자리 번호를 부여한다. (예: `0007-...`)
2. `docs/design-notes/template.md` 를 복사해 `NNNN-<kebab-제목>.md` 로 만든다.
3. 아래 섹션을 채운다 — 특히 **대안**과 **트레이드오프**를 비워두지 않는다 (여기가 핵심).
4. 가능하면 관련 코드(특히 ADR의 '영향'에 적은 핵심 파일) 상단에
   `// 설계 의도: ... (see docs/design-notes/NNNN-*.md)` 로 역참조를 건다. (양방향 추적)
5. `docs/design-notes/README.md` 인덱스에 한 줄 추가한다.

## 양식
```md
# NNNN. <결정 제목>

- 상태: 채택(Accepted) | 대체됨(Superseded by NNNN) | 폐기(Deprecated)
- 날짜: YYYY-MM-DD
- 맥락(앱/모듈): products | signup | shared

## 배경 / 문제
무엇을 해결하려 했는가. 어떤 제약(스펙·스택·시간)이 있었는가.

## 결정
무엇을 택했는가 (한 문단).

## 고려한 대안
- 대안 A — 장점 / 단점 / 탈락 이유
- 대안 B — ...

## 트레이드오프
이 결정으로 무엇을 얻고 무엇을 포기했는가.

## 영향
영향받는 파일·후속 작업·확장 시 고려점.
```

## 원칙
- "왜"를 적는다. "무엇"은 코드가 이미 말한다.
- 솔직하게 단점도 적는다. 트레이드오프를 인지했음을 보이는 것이 신뢰를 준다.
- 결정이 바뀌면 기존 ADR을 지우지 말고 상태를 `Superseded`로 바꾸고 새 ADR을 만든다.
