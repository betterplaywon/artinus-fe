# 0002. 로그인 식별자 — 이메일 또는 아이디 허용

- 상태: 채택(Accepted)
- 날짜: 2026-06-24
- 맥락(앱/모듈): login

## 배경 / 문제
로그인 식별자를 무엇으로 받을지 정해야 한다. 서비스마다 이메일/아이디/둘 다를 쓴다.

## 결정
**한 입력란에서 이메일 또는 아이디 둘 다 허용**한다. zod `refine` 으로 "유효한 이메일 형식 **또는**
유효한 아이디 형식(영문/숫자/밑줄 4–20자)"을 검증한다.

## 고려한 대안
- **이메일만** — 검증 단순하나 아이디 로그인 사용자를 배제.
- **아이디만** — 이메일 사용자를 배제.
- **탭/토글로 모드 분리** — 명확하나 단일 로그인 폼엔 과한 UI. 한 입력란 분기 검증이 더 가볍다.

## 트레이드오프
- 얻음: 사용자 입력 유연성, 단일 입력란의 단순한 UX.
- 포기: 검증 로직이 약간 복잡(형식 분기). 서버(여기선 MSW)도 두 형식을 모두 식별해야 한다.

## 영향
- `src/features/login/schema.ts`(`isEmail`/`isUsername` + `loginSchema`), `LoginForm`, `mocks/handlers.ts`(테스트 계정 2종).
